# API Documentation

While my primary focus is on product and internal documentation, I am comfortable working within technical environments to verify data. I use API tools to ensure my documentation accurately reflects how the system behaves "under the hood."

## :simple-swagger: REST API, OpenAPI, Swagger

On a regular basis, I use Swagger UI to explore the endpoints of the projects I document. I am familiar with:

* **Testing**: Authenticating and sending requests to verify real-world responses.
* **Schema Improvement**: As part of my professional development, I have experience enhancing OpenAPI schemas by adding descriptive titles, summaries, and clear examples to make them more developer-friendly.

## :material-graphql: GraphQL

I have a general understanding of GraphQL and how it differs from the REST API.

As part of my regular duties on the project, I am required to mention the GraphQL endpoints and parameters, which I do by reviewing the GraphiQL auto-generated document that lists all parameters. So far, improving this representation is outside of my responsibilities, so I only look up specific names to mention them in technical requirements or in product documents.
